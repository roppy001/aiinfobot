# Discord AIニュース定時レポートBot 要件定義書

## 1. 概要
毎朝9:00(JST)に、AI関連のニュース記事・指定サイトの更新情報を収集し、OpenAI APIで要約・重要度選定したレポートをDiscordチャンネルにWebhook経由で投稿する。AWS Lambda（Python）+ Amazon EventBridge（定期実行）で構成する。個人利用を前提とする。論文調査（arXiv等）は本要件では対象外とする。

## 2. 全体構成
```
EventBridge (cron, 0:00 UTC = 9:00 JST)
   → Lambda関数 (Python)
        1. 情報収集（RSS/ニュースAPI, 指定URL群）
        2. OpenAI APIで要約・重要記事選定（5〜10件）
        3. レポート整形
        4. Discord Webhookへ投稿
        （失敗時）→ Discordへエラー通知
```

## 3. 機能要件

### 3.1 情報収集
| 項目 | 内容 |
|---|---|
| ニュースRSS/API | **TechCrunch (AI)**、**ITmedia AI+** |
| 指定サイトURL群（AI企業・研究機関） | **OpenAI**、**Anthropic**、**Google**（DeepMind Blog / Google AI Blog等、詳細URLは実装時に確定）、**Preferred Networks**、**Sakana AI** |
| 指定サイトURL群（大手SIer・コンサルのプレスリリース） | **NTTデータグループ**、**富士通**、**日立製作所**、**NEC**、**日本アイ・ビー・エム (IBM Japan)**、**アクセンチュア** |
| サイトリスト管理 | 上記サイトは設定ファイル（YAML/JSON、例：`sources.yaml`）で管理し、後から追加・変更可能な構造にする |
| 取得期間 | 前回実行時刻〜現在（基本は「前日分」を対象）。重複排除は行わないため、同じ記事が翌日以降に再掲される可能性は許容する |

### 3.2 要約・レポート生成
- **OpenAI API** を使用（GPT系モデル、例：`gpt-4o` や `gpt-4o-mini` 等。モデルはコストと精度のバランスで後日確定）
- 収集した記事・論文群をOpenAI APIに渡し、**重要度が高い5〜10件を選定**した上で、各項目について日本語で要約（タイトル・出典・要約文・元記事URL）を生成
- レポート全体の冒頭に簡単なサマリー（その日のAI業界のトピック傾向など）を含める
- 出力形式はDiscord Embed、もしくは整形済みテキスト（文字数がDiscordの上限を超える場合は複数メッセージ・複数Embedに分割）

#### 3.2.1 重要度判定基準
利用者（社内でマーケティング・技術リードを兼務）の視点に合わせ、以下の観点をOpenAI APIへのプロンプトに組み込み、重要度をスコアリングした上で上位5〜10件を選定する。

| 観点 | 内容 | 重要度への影響 |
|---|---|---|
| ビジネス・マーケティングインパクト | 新製品/新サービスのローンチ、大型資金調達・M&A、大手企業でのAI導入事例、市場や競合構造に影響しうる発表 | 高：マーケティング視点で顧客説明・提案ネタになりうるものを優先 |
| 技術的重要性 | 新モデル・新技術のリリース、ベンチマークでの性能向上、OSS化、開発者向けAPI/SDKの重要な変更、アーキテクチャ上のブレークスルー | 高：技術リード視点で自社の技術選定・開発方針に影響しうるものを優先 |
| 競合・業界動向（指定SIer/コンサル） | NTTデータグループ、富士通、日立製作所、NEC、日本アイ・ビー・エム、アクセンチュアの生成AI関連プレスリリース | 中〜高：競合・業界内の動きとして原則ピックアップ対象に含める（内容が薄い定型リリースは除外可） |
| 情報源の一次性・信頼性 | OpenAI、Anthropic、Google、Preferred Networks、Sakana AI等の公式発表 | 高：一次情報源からの発表を優先し、二次報道・後追い記事より優先度を上げる |
| 新規性・重複排除（当日内） | 同一トピックを扱う複数記事がある場合は最も情報量の多い1件に集約 | 単純な後追い・広告・料金改定のみの告知は除外、または優先度を下げる |

- 上記基準はOpenAI APIへのプロンプト内で明文化し、各記事にスコア（例：ビジネス影響度・技術影響度をそれぞれ1〜5等）を付与させた上で、合計スコア順に上位5〜10件を選定する運用を想定する
- マーケティング・技術いずれかの観点で高スコアな記事は、両方の観点が低くても採用されるようにし、偏りを防ぐ（例：ビジネス影響度が低くても技術的ブレークスルーは採用する、など）

### 3.3 Discord投稿
- 投稿方式：**Webhook URL**（Botトークン方式は採用しない。将来的な拡張が必要になった場合に再検討）
- Webhook URLは環境変数で管理（3.4節参照）
- 投稿先チャンネルは1チャンネル固定（複数チャンネル対応は将来拡張）

### 3.4 実行トリガー
- Amazon EventBridge Scheduler（またはEventBridge Rule）で `cron(0 0 * * ? *)`（UTC 0:00 = JST 9:00）を設定
- 手動実行（テスト用）も可能なようにLambdaは外部トリガーに依存しないハンドラ設計とする

### 3.5 エラーハンドリング
- 情報収集・OpenAI API呼び出し・Discord投稿のいずれかで失敗した場合、**エラー内容を要約したメッセージをDiscordに通知**（同一Webhook、または専用の管理者向けWebhook/チャンネルを検討）
- CloudWatch Logsにも詳細エラーを出力（ログにAPIキー・Webhook URLは出力しない）
- Lambdaのタイムアウト・リトライ設定（EventBridge起因の自動リトライは基本オフにし、二重投稿を防止）

## 4. 非機能要件
| 項目 | 内容 |
|---|---|
| 実行環境 | AWS Lambda（Python 3.12等の最新安定版） |
| 実行時間 | 情報収集〜要約〜投稿までを想定し、タイムアウトは3〜5分程度で設定（要検証） |
| メモリ | 256〜512MB程度を想定（要検証） |
| シークレット管理 | Lambdaの環境変数（`OPENAI_API_KEY`, `DISCORD_WEBHOOK_URL`）に平文で設定する。**個人利用のためSecrets Manager／Parameter Storeは使用せず、リスクは承知の上で簡易運用とする**。`.env`やコード内へのハードコードは行わず、必ずLambdaの環境変数機能経由で設定する。リポジトリにAPIキー等を誤ってコミットしないよう`.gitignore`を徹底する |
| ログ・監視 | CloudWatch Logsに実行ログを出力 |
| コスト | Lambda実行課金＋OpenAI API従量課金。1日1回実行のため低コスト想定 |
| デプロイ | IaC（SAM/CDK/Terraform等）でLambda・EventBridge・IAMロールを管理することを推奨（ツール未確定、要相談） |

## 5. データ設計
- 重複排除は行わないため、**永続ストア（DynamoDB等）は不要**（ステートレスなLambda）
- サイト・RSSリストなどの設定は、Lambdaデプロイパッケージ内の設定ファイル（例：`sources.yaml`）で管理し、コード変更なしでの追加を可能にする

## 6. 未確定・今後決めるべき事項
- 各指定サイトの具体的な取得先URL・RSS有無の確認（プレスリリースページはRSSがない場合スクレイピング実装が必要）
- OpenAI APIに渡すプロンプトの具体的な文面設計（3.2.1の重要度判定基準を踏まえたスコアリング方式・要約の粒度やトーン）、利用モデルの確定
- IaCツール（SAM/CDK/Terraform）の選定
- エラー通知先を通常レポートと同一チャンネルにするか、別チャンネルにするか
